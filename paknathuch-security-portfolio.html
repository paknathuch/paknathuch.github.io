CVE-2025-41232
Spring Security — Authorization Bypass on Private Method Annotations
Critical CVSS 9.1 Spring Security ✔ Resolved

What Is This?

Disclosed 19 May 2025, this vulnerability exists in Spring Security Aspects. When @EnableMethodSecurity(mode=ASPECTJ) is active, the AspectJ weaver fails to locate and enforce method-security annotations (@PreAuthorize, @PostAuthorize, @Secured) placed on private methods. As a result, those private methods can be invoked without satisfying their declared authorization rules — a protection-mechanism failure (CWE-693).

How to Fix (Spring Boot 3)

Upgrade spring-security-bom to 6.4.6 or later in your pom.xml / build.gradle. Spring Boot 3.4.x manages this transitively via the BOM — bump the Boot parent version to pick it up automatically.

If you are using @EnableMethodSecurity(mode=ASPECTJ), migrate security annotations from private methods to package-private, protected, or public methods. AspectJ cannot weave private methods; the fix also enforces this at the framework level.

Verify the fix by running your security integration tests. Confirm that a call to a formerly-private annotated method without the correct role now throws AccessDeniedException.

pom.xml Snippet

<!-- Spring Boot parent – pulls Spring Security 6.4.6+ --> <parent> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-parent</artifactId> <version>3.4.6</version> </parent> <!-- Or pin directly if not using Boot parent --> <dependency> <groupId>org.springframework.security</groupId> <artifactId>spring-security-aspects</artifactId> <version>6.4.6</version> </dependency>

CVE-2025-48174
BusyBox — Stack Overflow / Heap Corruption in Container Base Image
High BusyBox / Docker ✔ Resolved

What Is This?

BusyBox is the tiny Unix toolset bundled inside most Alpine Linux–based Docker images (including eclipse-temurin:21-jre-alpine commonly used in Spring Boot microservices). This vulnerability class — consistent with the well-documented CVE-2022-48174 pattern — describes a stack overflow in BusyBox ash shell (ash.c) that can lead to arbitrary code execution or denial of service when a crafted input is processed inside the container. In containerised Spring Boot apps the shell is typically used for readiness/liveness probe scripts and init scripts in Kubernetes.

Affected Conditions

How to Fix (Spring Boot 3 / Docker)

Update your Dockerfile base image to an Alpine version that ships BusyBox ≥ 1.36.1. For JRE 21 that means switching to eclipse-temurin:21-jre-alpine rebuilt on Alpine 3.19+ or the Temurin tag that pulls a patched layer.

Run docker build --no-cache to force a fresh pull of the base layer — cached layers may still contain the vulnerable BusyBox binary.

Add a Trivy / Grype scan step to your CI pipeline to catch future base-image CVEs before they reach SIT/UAT. Example: trivy image --exit-code 1 --severity HIGH,CRITICAL myimage:tag

Dockerfile Snippet

# Use a patched Alpine-based JRE image FROM eclipse-temurin:21.0.5_11-jre-alpine # Verify BusyBox version inside image RUN busybox --help | head -1 # Should output: BusyBox v1.36.1 or higher COPY target/app.jar /app/app.jar ENTRYPOINT ["java", "-jar", "/app/app.jar"]

CVE-2026-22732
Spring Security — HTTP Security Headers Silently Dropped
Medium CVSS 7.5 Spring Security ✔ Resolved

What Is This?

Spring Security writes security response headers lazily by default — it waits until just before the response is committed. However, when a Spring MVC controller writes to response.getOutputStream(), calls response.flushBuffer(), or sets Content-Length via setIntHeader(), the servlet container commits the response before Spring Security's HeaderWriterFilter can inject its headers. The result: critical headers such as Cache-Control, X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, and Content-Security-Policy are silently omitted — leaving the browser without its security guardrails.

How to Fix (Spring Boot 3)

Preferred — upgrade: bump Spring Boot to a version that pulls Spring Security 6.4.15 or later. This is the permanent fix.

Workaround (if you cannot upgrade immediately): force eager header writing via a SecurityFilterChain customisation:

@Configuration @EnableWebSecurity public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { http .headers(headers -> headers // Force headers to be written BEFORE response commits .frameOptions(frame -> frame.deny()) .contentTypeOptions(Customizer.withDefaults()) .httpStrictTransportSecurity(hsts -> hsts .includeSubDomains(true).maxAgeInSeconds(31536000)) ); return http.build(); } @Bean public HeaderWriterFilter headerWriterFilter(HeadersConfigurer.<?> config) { HeaderWriterFilter filter = new HeaderWriterFilter(config.getHeaderWriters()); filter.setShouldWriteHeadersEagerly(true); // <-- key fix return filter; } }

CVE-2026-31789
OpenSSL — Heap Buffer Overflow in X.509 Certificate Processing
High OpenSSL / Container ✔ Resolved

What Is This?

CVE-2026-31789 is a heap buffer overflow in OpenSSL that occurs when processing X.509 certificates containing oversized OCTET STRING values in extensions such as Subject Key Identifier (SKID) or Authority Key Identifier (AKID). The root cause is an integer multiplication overflow during buffer-size calculation on 32-bit platforms, which allocates an undersized buffer and subsequently corrupts the heap. Exploitation against a 64-bit system requires a crafted certificate chain delivered to a vulnerable TLS endpoint. In Spring Boot microservices this manifests through embedded HTTP clients (RestTemplate, WebClient, Feign) or mutual-TLS configurations that process untrusted certificates.

How to Fix (Spring Boot 3 / Container)

Update the container base image to one shipping a patched OpenSSL. Alpine-based images: pull an Alpine ≥ 3.19.x tag; Debian-based: pull Debian Bookworm with updated libssl.

Use a 64-bit JRE base image (e.g., eclipse-temurin:21-jre-jammy on amd64). The integer overflow path is only exploitable on 32-bit address spaces.

If your services call external TLS endpoints, add certificate validation limits in your HTTP client config to reject certificates with abnormally large extension values.

Cherry-pick the patched Helm chart values onto the monthly release branch, rebuild images, and redeploy. Verify the fix with openssl version inside the running pod.
